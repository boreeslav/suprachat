# Шифрование сообщений

## Обзор

Все текстовые сообщения хранятся на сервере в формате `E1:` + AES-GCM. Сервер не знает мастер-пароль и не может расшифровать переписку.

## Мастер-пароль

- Минимум **6 символов**. По умолчанию **совпадает с паролем входа** (галочка в настройках и на экране разблокировки).
- Отдельный мастер-пароль можно задать, сняв галочку «Мастер-пароль совпадает с паролем входа».
- При входе сначала пробуется расшифровка паролем входа; если не удалось — показывается экран мастер-пароля, галочка снимается.
- Не сохраняется на сервере в открытом виде (только salt + verifier + RSA public key).
- Пара RSA **детерминированно выводится** из мастер-пароля, salt и `userId` — на любом устройстве с тем же паролем получается тот же ключ (нужен `supra-webcrypto.bundle.js`).
- Хранение сессии разблокировки зависит от режима (см. ниже «Режимы работы»):
  - **secure (HTTPS):** мастер-пароль сохраняется в шифрованном виде в IndexedDB (`SupraSecureStore`) и **переживает закрытие браузера** — повторный ввод не нужен.
  - **legacy (HTTP):** мастер-пароль только в `sessionStorage` (стирается при закрытии вкладки/браузера).
- **Выход** удаляет сессию (включая персистентный `SupraSecureStore`) и локальный кэш ключей; для разблокировки снова вводится только мастер-пароль.
- Аккаунты, созданные до этой схемы (случайная RSA + без бэкапа), требуют однократного «Сбросить шифрование» или входа со старого устройства для синхронизации legacy-бэкапа.
- **Сброс мастер-пароля** (если забыли): удаляет ключи на сервере и в `localStorage`, создаёт новую пару RSA; старые сообщения в группах не читаются, пока участники не выдадут ключ заново; доп. пароли групп нужно ввести снова.

## Личные чаты

Ключ: HKDF от мастер-ключа и пары `userId` (оба участника выводят один и тот же ключ).

## Группы

1. **Автопароль** — генерируется при создании группы, для каждого участника шифруется RSA-OAEP и сохраняется в `data/chat-member-keys.json`. Рассылается через API при добавлении участников (админ с разблокированным клиентом).
2. **Дополнительный пароль** (опционально) — задаётся создателем, передаётся **вне** мессенджера. На устройстве хранится в `localStorage` в формате **`L1:`** (AES-GCM, ключ от мастер-пароля). После выхода нужно ввести снова; без него можно отправлять сообщения уровня **basic**. Комбинируется с автоключом (XOR производных байт).

## Локальный кэш (IndexedDB)

- Кэш сообщений (`MessengerCacheDB`, store `messages`) хранит текст и превью ответа в поле **`bodyEnc`** (`L1:`), не в открытом виде.
- Расшифровка возможна только при разблокированном мастер-пароле в текущей вкладке.
- Поиск по тексту сообщений в кэше **отключён** (пункт «Найти» в чате убран).
- При обновлении до версии кэша 4 старый незашифрованный кэш сообщений очищается.

## API

| Метод | Описание |
|-------|----------|
| `GET /api/encryption/status` | Настроено ли шифрование |
| `POST /api/encryption/setup` | Salt, verifier, publicKey, privateKeyBlob (зашифрованный PKCS#8) |
| `GET /api/encryption/private-key-backup` | Зашифрованная копия приватного ключа для нового устройства |
| `POST /api/encryption/public-keys` | Публичные ключи пользователей |
| `POST /api/encryption/group-keys` | Сохранить обёрнутые автопароли |
| `GET /api/encryption/group-keys/{chatId}` | Свой обёрнутый автопароль |
| `GET /api/encryption/group-keys/{chatId}/missing` | Участники без ключа |

## Сброс данных

Старые незашифрованные сообщения несовместимы. Для чистого старта удалите `data/messages.json` и связанные файлы или всю папку `data/`.

## Режимы работы (secure / legacy)

Приложение автоматически выбирает один из двух путей. Источник истины — `SupraEnv` (`wwwroot/messenger/supra-env.js`): `secure = isSecureContext && нативный crypto.subtle`. Тип также можно зафиксировать при деплое (`SM_DEPLOY_PROTOCOL = https | http | auto`).

| | **secure** (HTTPS / `localhost`) | **legacy** (HTTP по IP) |
|---|---|---|
| Web Crypto | нативный `crypto.subtle` (полифилл не грузится) | node-forge `supra-webcrypto.bundle.js` |
| Мастер-пароль | персистентно в IndexedDB (`SupraSecureStore`), переживает рестарт | только `sessionStorage` |
| Кеш статики | **Service Worker** `/sw.js` (stale-while-revalidate) | кэш скриптов в `localStorage` (`app-script-cache.js`) |

### `SupraSecureStore` (персистентный мастер-пароль)

- В IndexedDB хранится **неэкспортируемый** (`extractable:false`) AES-GCM `CryptoKey`. Браузер не отдаёт сырой материал ключа обратно в JS — даже при XSS его нельзя выгрузить, только использовать для encrypt/decrypt.
- Мастер-пароль сессии шифруется этим ключом (`{iv, ct}`) и сохраняется рядом. На старте `SupraAuthCrypto.restoreSession()` расшифровывает и гидрирует `sessionStorage`, поэтому остальной код работает без изменений.
- **Компромисс безопасности:** персистентность означает, что любой, кто получил доступ к разблокированному профилю браузера/устройству, читает переписку без ввода пароля; XSS в любой будущей сессии тоже сможет расшифровать данные (но не украсть сам ключ). Это сознательный размен удобства на риск; для повышенной защиты в будущем — гейт через WebAuthn PRF/passkey (биометрия/PIN устройства).

### Service Worker (`/sw.js`)

- Регистрируется только в secure-режиме. Кеширует по белому списку **только статику** (`/messenger/*.js|css`, vendor, html-оболочки, иконки/шрифты).
- `/api/*`, `/hubs/*`, server-sent events, аватары и загрузки — **network-only**, не кешируются.
- Версия кеша `sm-static-v<build>` подставляется деплоем; старый кеш удаляется на `activate`.

## HTTP и доступ по IP

Браузер не даёт нативный `crypto.subtle` на **HTTP** с origin вида `http://192.168.x.x`. Для LAN без TLS подключён JS-полифилл:

- `wwwroot/messenger/vendor/supra-webcrypto.bundle.js` (node-forge, SubtleCrypto-совместимый API)
- Подключается **перед** `supra-crypto.js` на `login.html`, `register.html`, `index.html`
- Сборка входит в **`deploy.cmd` / `deploy\deploy.ps1`** (шаг «Build Web Crypto bundle»)
- Вручную: `src\SuperMessenger.Web\wwwroot\messenger\build-webcrypto.ps1`

На HTTPS или `http://localhost` по возможности используется нативный Web Crypto (полифилл не подменяет уже доступный `subtle`).

## Ограничения (MVP)

- Файлы/вложения в `data/uploads/` пока не шифруются (только текст сообщения).
- Персистентный мастер-пароль (secure) и `sessionStorage` (legacy) уязвимы к XSS в рамках сессии. Для максимальной защиты — гейт через WebAuthn PRF/passkey (см. «Режимы работы»).
- Поиск по тексту на сервере недоступен.
