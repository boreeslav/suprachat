# План: шифрование для ботов и пер-групповой переключатель шифрования

> Статус: **черновик плана, ожидает акцепта**. Реализация ведётся по частям, каждая часть принимается отдельно.

## 1. Цель

1. Дать боту **два режима общения**, работающих параллельно:
   - **Открытый** (как сейчас) — plaintext по REST `/api/bot-api/*` и WS `/ws/bot`.
   - **Приватный** — текстовые сообщения шифруются так же, как у пользователей (`E1:` + AES-GCM, RSA-OAEP для обмена ключами).
2. **Медиа (файлы/вложения) и mini-app канал остаются по открытому протоколу** — не шифруются (осознанное ограничение, как и для пользователей сейчас).
3. **Вернуть группам приватность**: добавить **переключатель шифрования в самой группе** (per-chat), который активирует протокол шифрования для этой группы.
4. Бот **может** использовать шифрование в личных чатах; в **зашифрованной группе обязан** его использовать.
5. **Цель безопасности:** сервер не хранит переписку в открытом виде. Пользователи — и допущенные в группы боты — держат свои пароли/ключи **под собственной ответственностью**.

## 2. Текущее состояние (точки интеграции)

### 2.1 Криптосхема (есть, только для людей)
- Мастер-пароль → PBKDF2 (310k) → мастер-ключ. RSA-пара **детерминированно** выводится из мастер-пароля + salt + `userId` (`supra-crypto.js`, `supra-webcrypto.bundle.js`).
- Личные чаты: ключ = HKDF от мастер-ключа и пары `userId`.
- Группы: автопароль, обёрнутый RSA-OAEP под публичный ключ каждого участника, хранится в `data/chat-member-keys.json` (`SupraChatMemberKeyRecord`). Доп. пароль группы (tier `protected`) — опционально.
- Шифротекст помечается префиксом `E1:`; сервер — zero-knowledge relay (`SupraMessengerService.IsEncryptedPayload`).
- Эндпоинты: `Controllers/EncryptionController.cs` (`status`, `setup`, `private-key-backup`, `public-keys`, `group-keys`, `reset`) — все `[Authorize]` (cookie).

### 2.2 Групповое шифрование = ГЛОБАЛЬНЫЙ флаг
- `AppAppearanceSettings.EnableGroupEncryption` (`Core/Entities/AppAppearanceSettings.cs`), задаётся в `AdminController`, отдаётся клиенту через `AppAppearanceService` (`enableGroupEncryption`).
- Клиент: `MessengerThemeManager.#adminEnableGroupEncryption` / `isAdminGroupEncryptionEnabled()`. От него зависит вся групповая крипта в `supra-messenger.js` (`#isPlaintextChatId`, `#ensureGroupChatEncryptionKeys`, `#setupGroupKeys`, распространение ключей).
- **Нет** пер-группового переключателя.

### 2.3 Боты = всегда plaintext
- Клиент: `#isBotChatId(chatId)` → `#isPlaintextChatId` = true (чат бота никогда не шифруется).
- Сервер `Infrastructure/Services/BotApiService.cs`:
  - `sendMessage`: отклоняет `E1:` («Боты отправляют только незашифрованные сообщения»).
  - `editMessage`: отклоняет `E1:`.
  - `RecordIncomingMessageAsync`: пропускает (не кладёт в инбокс) сообщения `IsEncryptedPayload`.
  - Сообщения бота сохраняются с `encryptionTier: "basic"` (plaintext).
- Транспорт бота: приём — WS `/ws/bot`; отправка — REST `/api/bot-api/*`; аутентификация `login`+`token` (SHA-256 hash, `BotApiService.AuthenticateAsync`).
- У бот-пользователя не заполнены `EncryptionPublicKey` / `EncryptionPrivateKeyBlob`.

### 2.4 Хранилище
- `FileDataStore` (JSON-файлы в `data/`). Сообщения — `SupraChatMessageRecord.Text` (plaintext или `E1:`), поле `EncryptionTier`. Инбокс бота — `BotInboxMessageRecord` (хранится 1 сутки). Файлы — `uploads/` (не шифруются).

### 2.5 CursorBot (клиент бота)
- `src/supra-bot-api.ts` — единственный сетевой клиент (WS + REST). `src/message-handler.ts` — логика. `src/config.ts` — env (`SUPRA_BASE_URL`, `SUPRA_BOT_LOGIN`, `SUPRA_BOT_TOKEN`). Node 20+ (есть `crypto.subtle`).

## 3. Целевая архитектура

### 3.1 Хранение ключей бота (под ответственностью владельца)
- Новый env-секрет `SUPRA_BOT_MASTER_PASSWORD` (в `CursorBot/.env`, уже в `.gitignore`).
- Бот при старте **детерминированно** выводит RSA-пару из мастер-пароля + salt + своего `botUserId` (тот же алгоритм, что у веб-клиента) и публикует на сервере: `EncryptionSalt`, `EncryptionVerifier`, `EncryptionPublicKey`, `EncryptionPrivateKeyBlob` через бот-аутентифицированный аналог `/api/encryption/setup`.
- Приватный ключ держится только у бота (выводится из пароля), сервер хранит лишь публичный ключ + клиент-зашифрованный blob — как у людей.

### 3.2 Два протокола параллельно
| Канал | Режим | Поведение |
|-------|-------|-----------|
| Личный чат бота | открытый (по умолчанию) или приватный | бот **может** шифровать текст |
| Группа без переключателя | открытый | как сейчас |
| Группа с переключателем шифрования | приватный | бот **обязан** шифровать текст |
| Медиа/файлы | открытый | всегда plaintext (вложения) |
| Mini-app канал (`/ws/mini-app`) | открытый | всегда plaintext |
| Каналы (channel) | открытый | как сейчас |

### 3.3 Пер-групповой переключатель
- Новое поле `SupraChatRecord.EncryptionEnabled` (bool, по умолчанию false → сохраняет текущее поведение групп).
- Переключатель в настройках группы (только админ). При включении: распространяются ключи всем участникам, **включая ботов** (RSA-OAEP врап автопароля под публичный ключ бота); новые сообщения требуют `E1:`.
- Клиентский гейт `isAdminGroupEncryptionEnabled()` дополняется/заменяется пер-чатовым `chat.encryptionEnabled` (глобальный флаг остаётся как «разрешено в принципе», решение — на уровне группы).

## 4. Разбивка на части (этапы с раздельным акцептом)

> Каждый этап: код → сборка/линт → краткий отчёт → **акцепт** → коммит этапа.

### Этап 0 — База и план (текущий)
- Сохранить этот план в `docs/`.
- Коммит + пуш текущего рабочего состояния и плана (без личных данных/секретов; `.env`, `bot-config.json`, `data/`, deploy-секреты уже в `.gitignore`).

### Этап 1 — Криптомодуль и ключи бота (сервер + CursorBot)
- Сервер: бот-аутентифицированный доступ к крипто-операциям (новые методы в `/api/bot-api/*`: `encryptionSetup`, `getPublicKeys`, `getGroupKey`) поверх существующей `SupraEncryptionService`, не трогая cookie-`EncryptionController`.
- CursorBot: env `SUPRA_BOT_MASTER_PASSWORD`; новый модуль `src/crypto/` (PBKDF2, детерминированная RSA-пара, AES-GCM `E1:`, HKDF, RSA-OAEP) — порт `supra-crypto.js` на Node WebCrypto.
- На старте бот публикует свой публичный ключ.
- **Без изменения поведения сообщений** (только инфраструктура ключей).

### Этап 2 — Приватные личные чаты бота
- Сервер: условно разрешить `E1:` от бота в `sendMessage`/`editMessage` для личных чатов; доставлять `E1:` в инбокс/WS бота (снять пропуск в `RecordIncomingMessageAsync` для зашифрованного пути).
- CursorBot: для личного чата вывести HKDF-ключ с пользователем, шифровать исходящий текст, расшифровывать входящий. Медиа/кнопки/mini-app — без шифрования.
- Открытый режим сохраняется (бот может слать plaintext, если у чата нет шифрования).

### Этап 3 — Пер-групповой переключатель (сервер + модель)
- Добавить `SupraChatRecord.EncryptionEnabled` + дефолт/миграция (старые группы = false).
- API мессенджера: метод включения/выключения шифрования группы (админ).
- Бот-API: `getChatInfo` отдаёт `encryptionEnabled`; `getMessages`/WS помечают зашифрованные группы.
- Требовать `E1:` для новых сообщений в зашифрованной группе.

### Этап 4 — Бот в зашифрованной группе (обязательное шифрование)
- Сервер: при включении шифрования группы — врап автопароля под публичный ключ бота-участника; бот-аутентифицированный `getGroupKey`; доставка `E1:` групповых сообщений в инбокс бота; релакс guard'ов для зашифрованных групп.
- CursorBot: получить и развернуть свой автопароль группы, шифровать/расшифровывать групповой текст; при отсутствии ключа — понятная ошибка.

### Этап 5 — Веб-UI пер-группового переключателя
- Переключатель «Шифрование группы» в профиле/настройках группы (админ).
- При включении: распространить ключи всем участникам, **включая бота**; предупредить, если у участника/бота нет опубликованного ключа.
- Привязать `#isPlaintextChatId`, `#shouldEncryptChat`, `#ensureGroupChatEncryptionKeys` к пер-чатовому флагу.

### Этап 6 — Документация, миграция, проверки
- Обновить `05-encryption.md` (бот как участник E2EE, пер-групповой переключатель) и `07-bot-api.md` (приватный режим, обязательность в зашифрованных группах, что медиа/mini-app остаются открытыми).
- Сценарии проверки: личный приватный чат, включение шифрования группы с ботом, миграция старых групп, fallback на открытый режим.

## 5. Влияние и риски
- Бот всё равно расшифровывает у себя → содержимое попадает в процесс бота/агента. Шифрование добавляет бота в круг доверия, но не скрывает переписку **от самого бота**.
- Управление мастер-паролем бота — новый класс секретов (env). Утечка = доступ к переписке бота.
- Инбокс: для зашифрованного пути сервер хранит `E1:`; догрузка `getMessages` отдаёт шифротекст, бот расшифровывает.
- Совместимость: дефолт `EncryptionEnabled=false` сохраняет текущее поведение существующих групп.
- Медиа и mini-app остаются открытыми — это явно зафиксированная граница, не дыра «по недосмотру».
